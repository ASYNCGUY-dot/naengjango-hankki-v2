## 1. 요약

| 지표 | 값 |

| 화면 기준 완성도 (14개 화면 가중 환산) | **약 54%** |
| 완전히 이식된 화면 | 4 / 14 |

## 3. 백엔드 로직 재사용 현황

**API로 연결된 V1 에이전트 (9개)**
- `auth` — 엔드포인트만 존재, 프론트 미사용
- `profile` — 온보딩
- `pantry` — 내 냉장고
- `ingredient_catalog` / `ingredient_favorite` — 재료 찾아보기
- `recommendation` — 추천 (이유 생성 단계 제외)
- `substitution` — 대체재료
- `safety` — 회수·유통기한 확인
- `favorite` — 레시피 즐겨찾기
- `review` — 후기 + AI 요약
- `popular_recipe` / `youtube` — 인기 영상

**파일은 있지만 API 미연결 (9개)**
- `price_agent` — KAMIS 가격 등급
- `nutrition_target_agent` — 개인 맞춤 영양 분석
- `recommendation_graph` — LangGraph 기반 추천 이유
- `user_recipe_agent` — 유저 레시피 등록/승인
- `ingredient_submission_agent` — 재료 등록 제안
- `like_agent` — 좋아요
- `notify_agent` — 알림
- `portion_agent` — 인분수 조절
- `seasonal_agent` / `shopping_agent` — 제철 재료, 장보기 리스트

## 4. V2에서 새로 생긴 것 (V1에는 없던 부분)

- **PWA 지원**(매니페스트·서비스워커·홈 화면 설치)
- **추천 API N+1 쿼리 개선**(배치 조회 + 인덱스)
- **배포 준비**(Render 설정, API_BASE 환경변수화)
- **로고/디자인 목업**(헤더·하단 탭·카드 스타일)

V1 회고에서 지적된 "Streamlit 구조상 UI 자유도·성능·상태관리 한계"를 해소하는 방향의 작업은 계획대로 진행 중이다.

## 5. 다음 스텝 권장 — V1 퀄리티를 넘기 위한 4가지 케이스

V1을 "넘어서는" 기준을 두 축으로 본다: ① V1에 있던 차별점(멀티 LLM 추천이유, 가격·맞춤영양, 로그인)을 최소한 회복하는 것, ② V1이 구조적으로 못했던 것(테스트, DB 확장성, 성능)을 실제로 해결하는 것. 목표에 따라 케이스를 골라 진행하면 된다.

### Case A — 핵심 차별점 복원 (V1의 "가장 자랑스러운 기능"부터)
현재 백엔드 파일은 있는데 API에 안 붙은 것들이라, 상대적으로 빠르게 갚을 수 있는 기술 부채다.
1. **로그인/세션 영속화** — `auth` 라우터는 이미 있으니 Reflex 프론트에 로그인 화면을 붙이고, `rx.Cookie`/`rx.LocalStorage`로 `user_id` 유지 (지금은 새로고침하면 날아감 — 가장 눈에 띄는 결함)
2. **AI 추천 이유(LangGraph, 멀티 LLM)** — `recommendation_graph.py`를 `recommendation` 라우터에 연결, 레시피 상세에 "왜 추천했는지" 노출
3. **맞춤 영양분석** — `nutrition_target_agent`를 붙여서 "탄단지나트륨" 단순 표시를 "내 목표 대비" 비교로 격상
4. **KAMIS 가격 등급** — `price_agent` 연결, 레시피 상세에 예상 재료비 노출

### Case B — 누락 화면 채우기 (기능 커버리지 parity)
V1엔 있었는데 V2엔 화면 자체가 없는 것들.
1. **레시피 찾아보기** — 검색+카테고리 필터, 추천에 의존하지 않고 전체 레시피 탐색
2. **레시피 등록 폼** — `user_recipe_agent` 연결, 유저 출처 다양화(V1 로드맵에도 있던 항목)
3. **내 냉장고 디테일** — 유통기한 임박 배지, 재료별 영양/가격 직접 등록
4. **재료 등록 제안** — `ingredient_submission_agent` 연결

### Case C — V1이 구조적으로 못 했던 것 (진짜 서비스급으로)
V1 회고 문서(`발표자료_한계와다음단계.md`)에 "알고 있는 한계"로 명시된 항목들 — 여기서 이기면 확실한 우위다.
1. **자동화 테스트 도입** — V1은 "수동 점검 + CI 문법체크"뿐이었다. pytest로 agent 단위 테스트 + API 통합 테스트부터
2. **SQLite → PostgreSQL** — V1 로드맵에 있던 항목, `psycopg2-binary`는 이미 requirements.txt에 있음(미사용). 동시접속 확장성 확보
3. **재료 검색 성능** — 지금 30만 행 LIKE 검색 1.7초 걸리는 걸 인덱스/FTS로 개선 (이미 알려진 이슈)
4. **건강정보 암호화 + 개인정보처리방침** — V1 로드맵에 있었지만 미완이던 부분

### Case D — V1에 없던 것으로 신규 우위 확보
1. **PWA 완성도** — 지금은 매니페스트/서비스워커만 있는 수준. 오프라인 캐싱, 설치 후 앱처럼 동작하는지 실제 검증
2. **좋아요/알림** — `like_agent`, `notify_agent` 연결로 V1엔 없던 참여 기능 추가
3. **LangGraph 전체 확장** — V1도 인정한 부분(추천-이유 단계에만 적용), 8개 에이전트 전체로 확장

### 추천 순서
Case A(로그인 세션 + AI 추천이유)를 가장 먼저 권한다. 지금 V2는 새로고침하면 로그인 정보가 날아가는 상태라 데모조차 불안정하고, AI 추천 이유는 V1 발표자료에서 가장 먼저 내세운 차별점이라 이게 빠진 채로는 "V1보다 낫다"고 말하기 어렵다. 그다음 Case C의 테스트/성능(구조적 약점 해소)로 넘어가면 실제로 V1을 능가하는 지점이 생긴다.
